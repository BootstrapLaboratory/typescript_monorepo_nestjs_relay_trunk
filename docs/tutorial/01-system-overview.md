# System Overview

This project is a Rush-managed TypeScript monorepo for a GraphQL chat
application. The interesting design is not just the app. It is the way runtime
code, generated contracts, delivery metadata, provider automation, and CI are
kept in separate layers.

That separation lets each part change at its own pace. A server resolver can
change without touching Cloudflare provisioning. A deploy scenario can gain a
new provider step without changing React components. A GraphQL schema change is
visible to both backend and frontend through a committed contract.

## The Main Boundaries

| Boundary                      | Path                                                          | Owns                                                                                                  |
| ----------------------------- | ------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Backend app                   | `apps/server`                                                 | Runtime API behavior, GraphQL schema generation, database access, migrations, identity, subscriptions |
| Frontend app                  | `apps/webapp`                                                 | Browser UI, routing, Relay operations, frontend auth state, static build output                       |
| API contract                  | `libs/api`                                                    | Committed `schema.gql` generated by the server and consumed by the webapp                             |
| Rush config                   | `rush.json`, `common/config/rush`                             | Project membership, dependency installation, lockfile ownership, bulk commands                        |
| Rush Delivery metadata        | `.dagger`                                                     | Package targets, deploy targets, deploy ordering, validation targets, cache and toolchain providers   |
| Provider scripts              | `deploy/cloudrun`, `deploy/cloudflare-pages`                  | Concrete deployment scripts for production targets                                                    |
| Provider packages             | `deploy/providers/*`                                          | Typed provider functions used by preparation scenarios                                                |
| Scenario engine and scenarios | `deploy/scenario-engine`, `deploy/scenarios`, `deploy/wizard` | Guided pre-deployment setup, resumable state, secret-safe handoffs                                    |
| CI                            | `.github/workflows`                                           | Human-facing triggers and cloud authentication before handing control to Rush Delivery                |

The consequence is that each layer has a clear owner. When a change crosses a
boundary, the crossing is intentional: package dependencies, generated schema,
environment variables, deploy metadata, or scenario outputs.

## Runtime Flow

At runtime, the webapp talks to the server through GraphQL:

```text
browser
  -> React routes and Relay operations
  -> GraphQL HTTP requests and graphql-ws subscriptions
  -> NestJS GraphQL resolvers
  -> services, database, and Redis pub/sub
```

The server is the source of truth for the GraphQL schema. The webapp does not
invent a parallel API model; it compiles Relay operations against the committed
schema in `libs/api/schema.gql`.

That gives the project a practical contract workflow:

1. Backend code-first GraphQL types produce `schema.gql`.
2. The schema is committed in `libs/api`.
3. The webapp depends on `api-contract`.
4. Relay codegen validates frontend operations against that schema.

The benefit is early failure. A broken schema or invalid Relay operation fails
during verification instead of becoming a production browser error.

## Release Flow

The release path is intentionally not scattered across GitHub Actions jobs.
GitHub Actions authenticates and passes inputs, then Rush Delivery runs the
release model:

```text
GitHub Actions
  -> Rush Delivery
  -> detect affected targets
  -> validate
  -> build
  -> package
  -> deploy according to the deploy mesh
```

The deploy targets are:

- `server`, packaged through Rush deploy and deployed to Cloud Run
- `webapp`, packaged as the Vite `dist` directory and deployed to Cloudflare
  Pages

The deploy mesh says `webapp` depends on `server`. That does not mean the
frontend imports backend code. It means the release order recognizes an
operational relationship: frontend endpoint values and CORS behavior must agree
with the backend deployment.

## Preparation Flow

Production setup has a different shape from production deployment. The scenario
system prepares infrastructure and repository configuration before the CI
release workflow runs.

The current production scenario:

```text
Cloud Run + Cloudflare Pages + Neon + Upstash
```

does this kind of work:

- prepare Cloud Run prerequisites in an existing billing-enabled Google Cloud
  project
- collect Neon and Upstash URLs as transient secrets
- sync runtime secrets into Google Secret Manager
- prepare a Cloudflare Pages project
- configure GitHub repository variables and secrets

It does not trigger deployment automatically. That is deliberate. Preparation
and deployment have different safety profiles. Preparation may need human input
and secret collection; deployment should happen through the repeatable CI
release path.

## Why The Project Is Split This Way

The design avoids three common problems.

First, it avoids app code becoming deployment code. NestJS modules and React
components should not know how to create Google service accounts or configure
Cloudflare Pages.

Second, it avoids CI becoming the architecture. GitHub Actions remains a thin
entrypoint instead of a large procedural release script.

Third, it avoids generated contracts becoming invisible. The GraphQL schema is
a real Rush project artifact, so server and webapp changes meet at a committed
file.

The cost is that the repository has more explicit metadata and more named
boundaries. That is an acceptable tradeoff here because the project is meant to
be deployable, repeatable, and understandable from source control.
